Quant analysis workflow

Rev. 9/24/2024 LRM

Overview

This document summarizes the basic process for performing quant analyses on the microprobe as is often done by geologists. This is the general process:

  1. Make a plan
  2. Create a quantitative analysis recipe
  3. Analyze standards to generate calibration files
  4. Validate your recipe
  5. Put in points and analyze your samples
  6. Quality control and postprocessing

Make a plan

Before you do anything on the microprobe, it's a good idea to form a plan about how you want to perform your analyses. This plan should include the following information:

  • What are the questions that you hope to answer by collecting data on the microprobe?
  • What material will you be analyzing?
  • Which elements are you interested in, and roughly what concentrations are you expecting?
  • What spectrometer configuration do you plan to use (which spectrometer/which element/which crystal)
  • How long do you expect/need your analyses to take? (count times)
  • What beam conditions do you plan to use? (Accelerating voltage, beam current, spot size)
  • What reference materials do you plan to use to calibrate your recipe?
  • How do you plan to analyze your standards? (Same or different conditions from unknowns?)
  • What reference materials do you plan to use to validate your recipe?
  • What metrics do you plan to use for quality control?

Most of this information can be found by reviewing the literature to determine "what do other people do," but it is expected that few microprobe users will be prepared with all of this information in advance, so please ask the lab manager for help when preparing this plan!

Creating a quantitative analysis recipe

In our lab, we use JEOL's "PC-EPMA" program for data acquisition. This program uses a "recipe" template to store all of the settings, or "Conditions" associated with the analyses that we do. Setting up a recipe forces the user to finalize the plan so that it can be executed by the microprobe's automation software.

Here is the SOP document for creating a new quantitative analysis recipe: SOP -- WDS quantitative analyses -- new recipe

Analyze standards to generate calibration files

When you are setting up your quantitative analysis recipe, if you examine one of your elements in the orange "Analysis Element Condition" section, you will see that the "Standard" field says "Cal-Std" by default. This is where you specify which standard file is used to relate the X-ray count rate obtained during the analysis to an elemental/oxide concentration. The "Cal-Std" setting refers to a factory-calibrated, or "standardless", analysis mode which can be used to perform analyses without doing any calibrations in advance -- this is a great tool for training users, testing out/debugging new recipes, and the results are usually not too terrible (accurate to within a few wt%). When we calibrate the microprobe, we analyze a reference material with a known composition using a "Standard Analysis" recipe (as opposed to your "Quantitative Analysis" recipe). This creates a standard analysis file that will be used instead of the default factory calibration.

Validate your recipe

Here is the SOP document for performing quantitative analyses using an existing recipe: SOP -- WDS quantitative analyses -- existing recipe

At this point, you have a calibrated quantitative analysis recipe, and you technically could start putting in points and analyzing your samples. However, before you do this, you should take some time to analyze one or a few reference materials for which you know the correct composition to make sure that your recipe is giving you accurate results. Also, this is when you will start to apply the "metrics for quality control" part of your analysis plan. For example, some users like to see analyses with totals between 98.5 to 101.0 wt%, and individual element/oxide concentrations that are accurate to within 5 percent, relative.

If you're not happy with the results, you probably made a mistake when running your standard analyses -- this happens all the time, and it is very easy for user error to occur during the calibration step. Try re-doing any of the affected calibrations to see if things improve. If they don't, ask the lab manager for assistance.

If you are happy with the results, then you can go ahead and start analyzing your samples.

Put in points and analyze your samples

There isn't much to say about this step because how and where you analyze your sample depends on your unique research goals. If you have any questions about this, the lab manager can help you. Otherwise, have fun putting in points!

...but wait! -- what are you planning to do about instrumental drift? A good way to check and correct for instrumental drift is to periodically analyze one or a few reference materials to make sure that their measured compositions aren't changing over time. I recommend analyzing these about once every hour or two. That means that, if your analyses take about three minutes to run, then every 20th or 40th point in your acquisition sequence should be an analysis of a reference material.

Ok, now you can have fun putting in your points. Please don't hesitate to call or text the lab manager if you get stuck or have any questions.

Quality control and postprocessing

Did your data turn out ok? Do you need to apply a drift correction? Do you have questions about how to export your data? The lab manager can help you with these questions.

In general, it is recommended that you use the "Summary" button to export your data to a comma-separated text file (spreadsheet) and then save this spreadsheet in the folder corresponding to the acquisition that ran when you clicked "Acquire" or whatever the case may be. For example, if the data are all saved in the folder "QNT_0001" then you should save your spreadsheet in that folder.

After you have exported your data, you can transfer the data from the microprobe using either our FileZilla server (from which it will be automatically archived), or you can also use Google Remote Desktop to do this if you are using the microprobe remotely.
